# 短剧剧本反推工作台设计说明

- 日期：2026-04-03
- 项目：`F:\local_code\novalai`
- 状态：已完成交互设计确认，待进入实现计划

## 1. 目标

为现有短剧剧本反推脚本链路提供一个本地可用的内部工作台，满足以下目标：

- 管理 `videos/` 下已有视频和 `output/` 下已有历史结果
- 支持页面上传新视频，并选择“长期保存”或“临时处理”
- 支持跑单个视频、批量入队、跑全部视频
- 在页面中展示阶段型进度：
  - `ASR`
  - `视频切片`
  - `多模态分析`
  - `剧本整合`
- 在一个工作台中查看视频、对白、分段结果和剧本结果
- 第一版优先本机单人使用，但结构上为后续局域网扩展预留空间

## 2. 非目标

第一版明确不包含以下能力：

- 多用户登录与权限管理
- 任务并行调度与多 worker
- 结果多版本并存与版本对比
- 人工打分、备注、状态标记
- 复杂实时通信方案，如 WebSocket 日志推送
- 重写 `video2script.py` 的核心处理链路

## 3. 总体方案

采用 `FastAPI + React(Vite) + 单进程任务队列` 的实现方式。

### 3.1 后端

使用 `FastAPI` 提供本地 HTTP 服务，职责如下：

- 接收上传视频
- 管理视频列表与历史输出
- 创建和维护任务队列
- 串行调用 `video2script.py`
- 解析脚本标准输出并映射为阶段状态
- 对前端提供视频、对白、分段、剧本等读取接口

### 3.2 前端

使用 `React + Vite` 实现内部工作台页面，职责如下：

- 上传视频与触发任务
- 浏览任务队列和历史结果
- 展示阶段型进度
- 播放视频并联动结果查看
- 查看 `dialogues.json`、`segments.json` 和 `script.txt`

### 3.3 处理引擎

`video2script.py` 继续作为既有处理引擎，不拆分为新系统。Web 后端通过子进程执行：

```bash
python video2script.py <video_path>
```

第一版通过解析标准输出获取阶段状态，不要求先改造核心推理逻辑。

## 4. 信息架构与界面设计

### 4.1 视觉定位

- 类型：内部生产工具
- 气质：戏剧感工作台
- 结构：`C 双栏指挥台`
- 调整：左侧减重，运行阶段细节改为可展开抽屉

### 4.2 页面结构

#### 顶部区域 `TopBar`

展示全局信息与批处理操作：

- 系统名称
- 任务统计
- `跑全部`
- `暂停队列`
- `重跑失败`
- ASR 状态
- 多模态服务状态

#### 左侧 `UploadPanel`

用于上传视频：

- 拖拽/选择视频
- 选择上传模式：
  - `长期保存`
  - `临时处理`

#### 左侧 `QueuePanel`

用于展示视频和任务队列：

- `videos/` 中已有视频
- 历史处理结果视频
- 当前排队任务
- 当前运行任务高亮
- 简状态：
  - `queued`
  - `running`
  - `completed`
  - `failed`

#### 左侧 `TaskDrawer`

折叠式任务进度抽屉，仅显示当前任务的详细阶段信息：

- `ASR`
- `视频切片`
- `多模态分析`
- `剧本整合`

默认收起，仅在需要盯运行过程时展开。

#### 右侧 `PreviewStage`

主舞台区域，页面视觉焦点：

- 视频播放器
- 当前片段/时间轴
- 当前剧本预览

默认优先展示“视频对照”。

#### 右侧 `ResultPanel`

切换查看不同结果产物：

- `对白`
- `分段`
- `剧本`

默认 Tab 为 `剧本`。后续如加入评分、备注、版本对比，仍在该区域扩展。

### 4.3 交互原则

- 左侧只负责调度和任务管理
- 右侧只负责视频和结果查看
- 上传、运行、查看三类动作不互相抢焦点
- 首页默认清爽，详细运行细节按需展开

## 5. 后端接口设计

### 5.1 健康检查

`GET /api/health`

返回：

- Web 服务状态
- ASR 服务可用性
- `videos/` 可访问性
- `output/` 可访问性

### 5.2 视频列表

`GET /api/videos`

返回统一视频记录，字段包括：

- `video_id`
- `video_name`
- `video_path`
- `has_output`
- `latest_task_status`
- `output_ready`

说明：

- `video_id` 为前后端交互使用的稳定标识
- `video_name` 为展示名称
- 后端为每个上传文件生成唯一存储名，避免同名视频导致结果目录冲突

### 5.3 上传视频

`POST /api/uploads`

参数：

- 视频文件
- `persist=true/false`

行为：

- `persist=true`：落到 `videos/`
- `persist=false`：落到 `tmp_uploads/`

补充规则：

- 上传后后端统一生成唯一存储文件名
- 前端展示原始文件名，但后端内部始终以 `video_id` 和唯一文件路径作为主键

### 5.4 创建任务

`POST /api/tasks`

参数：

- `video_id`
- 可选执行模式

行为：

- 任务入内存队列
- 若无任务运行则立即启动

### 5.5 跑全部

`POST /api/tasks/run-all`

行为：

- 将当前可处理视频批量入队

### 5.6 任务列表

`GET /api/tasks`

返回：

- 排队任务
- 运行中任务
- 已完成任务
- 失败任务

### 5.7 单任务详情

`GET /api/tasks/{task_id}`

返回：

- 当前阶段
- 当前阶段进度
- 最近日志
- 开始/结束时间
- 错误信息

### 5.8 结果聚合

`GET /api/results/{video_id}`

返回：

- `dialogues.json`
- `segments.json`
- `script.txt`
- 输出目录信息

### 5.9 视频流

`GET /api/media/{video_id}`

用于播放器直接加载视频。

## 6. 任务模型与执行流程

### 6.1 调度方式

第一版采用单 worker 串行执行：

- 所有任务进入一个内存队列
- 同一时刻只跑一个任务
- 当前任务结束后自动执行下一个

### 6.2 任务字段

每个任务维护以下核心字段：

- `task_id`
- `video_id`
- `video_name`
- `video_path`
- `source_type`
- `status`
- `stage`
- `stage_progress`
- `created_at`
- `started_at`
- `finished_at`
- `error_message`
- `logs_tail`

### 6.3 状态定义

任务状态：

- `queued`
- `running`
- `completed`
- `failed`

阶段状态：

- `queued`
- `asr`
- `segmenting`
- `multimodal`
- `merging`
- `done`
- `failed`

### 6.4 日志到阶段的映射

通过解析 `video2script.py` 标准输出映射阶段：

- 匹配 `[Step A]` 或 `调用 ASR 接口`：
  - `stage = asr`
- 匹配 `[Step B]`：
  - `stage = segmenting`
- 匹配 `视频分段完成计划，共 N 段`：
  - 记录总分段数
- 匹配 `生成视频片段 [i/N]`：
  - 仍属 `segmenting`
  - 更新阶段百分比
- 匹配 `分析视频片段 [i/N]`：
  - `stage = multimodal`
  - 更新阶段百分比
- 匹配 `[Step C]` 或 `整合剧本...`：
  - `stage = merging`
- 匹配 `剧本已保存:`：
  - `stage = done`
  - `status = completed`

### 6.5 前端状态获取方式

第一版使用轮询，不使用 WebSocket：

- `GET /api/tasks`：每 2 秒轮询
- `GET /api/tasks/{task_id}`：每 1 到 2 秒轮询
- 任务完成后刷新 `GET /api/results/{video_id}`

选择轮询的原因：

- 第一版更稳
- 更容易实现和调试
- 对本机单人使用已足够

## 7. 文件与落盘规则

### 7.1 长期保存上传

- 上传文件保存到 `videos/`
- 遇到重名文件时自动加时间戳或唯一后缀，不覆盖原文件
- 结果落到 `output/<video_name>/`

### 7.2 临时上传

- 文件保存到 `tmp_uploads/<task_id>/`
- 任务运行直接使用该路径
- 任务完成后保留结果
- 第一版默认保留临时原视频，方便复查和复跑

补充规则：

- 临时上传文件同样使用唯一存储文件名
- 为避免 `video2script.py` 按文件 stem 输出时发生目录覆盖，进入处理前即保证文件 stem 唯一

### 7.3 输出规则

沿用当前既有结构：

- `output/<video_name>/dialogues.json`
- `output/<video_name>/segments.json`
- `output/<video_name>/script.txt`

### 7.4 重跑规则

第一版重跑默认覆盖当前视频目录下的三个主结果文件，不做多版本保留。

## 8. 异常处理

第一版需覆盖以下场景：

- 上传失败：
  - 前端提示错误
  - 保留重试入口
- 任务执行失败：
  - 标记为 `failed`
  - 保留错误信息和日志尾部
- ASR 或多模态服务不可用：
  - 顶部状态栏标红
  - 任务详情显示失败阶段
- 结果文件不完整：
  - 右侧结果面板显示产物不完整提示
  - 已有产物仍可查看
- 当前任务失败：
  - 队列自动继续处理下一个任务

## 9. 实现范围

第一版必须实现：

1. 展示现有 `videos/` 和 `output/` 历史结果
2. 上传视频并选择“临时处理 / 长期保存”
3. 跑单个视频
4. 一键跑全部 / 批量入队
5. 展示阶段型进度
6. 查看视频、对白、分段、剧本
7. 查看失败状态和简要日志
8. 不影响现有命令行 `video2script.py` 使用方式

## 12. 当前结论

第一版的正确方向不是重写推理流程，而是把既有脚本链路包装成一个可视化、可调度、可查看的本地工作台。该方案以最小侵入方式提升可用性，并为后续扩展留下清晰边界。
